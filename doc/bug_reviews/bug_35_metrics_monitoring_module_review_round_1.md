## Finding 1: BigKey violates the Object.equals/hashCode contract

Severity: Medium

Files:
- `src/main/java/io/velo/monitor/BigKeyTopK.java:16-36`

Code excerpt:

```java
public record BigKey(String key, int length) {
    @Override
    public boolean equals(Object obj) {
        if (this == obj) {
            return true;
        }
        if (obj == null || getClass() != obj.getClass()) {
            return false;
        }
        BigKey that = (BigKey) obj;
        return key.equals(that.key);  // only compares key
    }
    // hashCode() is NOT overridden — auto-generated by record using BOTH key and length
}
```

Root cause:
`BigKey` is a Java record. Records auto-generate `hashCode()` based on all components (`key` and `length`). The class overrides `equals()` to compare only `key`, but does NOT override `hashCode()`. This violates the `Object.hashCode()` contract: *if two objects are equal according to `equals()`, they must have the same hash code*. Two `BigKey("foo", 10)` and `BigKey("foo", 20)` are `equals()` (same key) but produce different hash codes.

Impact:
Today, `BigKey` is used exclusively inside a `PriorityQueue`, which does not use `hashCode()`, so there is no runtime failure. However:

1. If any future code places `BigKey` in a `HashSet`, `HashMap`, or any hash-based structure, lookup will break: two equal keys can land in different hash buckets and never be found.
2. The contract violation makes the class unsafe for general-purpose use. A monitoring utility class is especially likely to be used in collections for aggregation or deduplication.
3. The existing test `BigKeyTopKTest` only tests `equals()` directly; it does not test `hashCode()`, so the contract violation is not caught by tests.

Suggested fix:
Override `hashCode()` to be consistent with `equals()`:

```java
@Override
public int hashCode() {
    return key.hashCode();
}
```

## Finding 2: initBigKeyTopK silently ignores DynConfig updates after first initialization

Severity: Medium

Files:
- `src/main/java/io/velo/persist/OneSlot.java:812-817`
- `src/main/java/io/velo/persist/OneSlot.java:198` (constructor call)
- `src/main/java/io/velo/persist/DynConfig.java:74-84` (SUPPORTED_ITEMS entry)
- `src/main/java/io/velo/persist/DynConfig.java:256-258` (switch case — dead code path for this key)
- `src/main/java/io/velo/persist/LocalPersist.java:204-218` (initialization order)

Code excerpts:

```java
// OneSlot constructor (line 198)
initBigKeyTopK(10);
```

```java
// OneSlot.java:812-817
void initBigKeyTopK(int k) {
    // may be dyn config init already init big key top k
    if (bigKeyTopK == null) {
        bigKeyTopK = new BigKeyTopK(k);
    }
}
```

```java
// DynConfig.java:74-84 — SUPPORTED_ITEMS entry
BigKeyTopK.KEY_IN_DYN_CONFIG, new DynConfigItem() {
    @Override
    public Object parseAndValidate(@NotNull Object value) {
        return parsePositiveInt(BigKeyTopK.KEY_IN_DYN_CONFIG, value);
    }

    @Override
    public void apply(short currentSlot, OneSlot oneSlot, @NotNull Object value) {
        oneSlot.initBigKeyTopK((int) value);
        log.warn("Global config for current slot set monitor_big_key_top_k={}, slot={}", value, currentSlot);
    }
},
```

```java
// LocalPersist.java:204-218
public void initSlotsAgainAfterMultiShardLoadedOrChanged() throws IOException {
    for (var oneSlot : oneSlots) {
        oneSlot.clearGlobalMetricsCollect();
    }
    var firstOneSlot = firstOneSlot();
    if (firstOneSlot != null) {
        firstOneSlot.addGlobalMetricsCollect();
        if (!ConfForGlobal.initDynConfigItems.isEmpty()) {
            for (var entry : ConfForGlobal.initDynConfigItems.entrySet()) {
                firstOneSlot.getDynConfig().update(entry.getKey(), entry.getValue());
            }
        }
    }
}
```

Root cause:
The initialization sequence is:

1. `new OneSlot()` constructor calls `initBigKeyTopK(10)` — creates `BigKeyTopK` with k=10, so `bigKeyTopK` is non-null.
2. Later, `LocalPersist.initSlotsAgainAfterMultiShardLoadedOrChanged()` calls `dynConfig.update(BigKeyTopK.KEY_IN_DYN_CONFIG, value)` which calls `DynConfigItem.apply()` → `oneSlot.initBigKeyTopK(value)`.
3. But `initBigKeyTopK()` checks `if (bigKeyTopK == null)` and skips because `bigKeyTopK` was already created in step 1.

The log message `"Global config for current slot set monitor_big_key_top_k=..."` is printed, giving the false impression that the config was applied. In reality, the config value is stored in the JSON file but never applied to the runtime `BigKeyTopK` instance. Runtime `CONFIG SET` or `manage slot N update-dyn-config` updates are also silently ignored for the same reason.

Impact:

1. Users cannot change `monitor_big_key_top_k` at runtime. A config change that says "track top 100 keys" will keep the default top 10.
2. The DynConfig `apply()` log message is misleading — it claims the config was applied when it was not.
3. If the intent was that the DynConfig value should only apply before construction (as the comment suggests: "may be dyn config init already init big key top k"), the code path through `SUPPORTED_ITEMS` and the switch case are dead code that adds confusion.

Suggested fix:
Replace the null-check guard with a replacement that always applies the new k value:

```java
void initBigKeyTopK(int k) {
    if (bigKeyTopK == null || bigKeyTopK.getQueue().size() != k) {
        bigKeyTopK = new BigKeyTopK(k);
    }
}
```

Or, if preserving existing tracking data is important when the k value has not changed, only recreate when k differs:

```java
void initBigKeyTopK(int k) {
    if (bigKeyTopK == null) {
        bigKeyTopK = new BigKeyTopK(k);
    } else if (this.k != k) {
        // k changed — rebuild with new capacity
        bigKeyTopK = new BigKeyTopK(k);
    }
}
```

Note: this requires storing the current `k` value. Alternatively, always recreate:

```java
void initBigKeyTopK(int k) {
    bigKeyTopK = new BigKeyTopK(k);
}
```

## Finding 3: RuntimeCpuCollector.close() is a misleading no-op

Severity: Low

Files:
- `src/main/java/io/velo/monitor/RuntimeCpuCollector.java:14-16`
- `src/main/java/io/velo/MultiWorkerServer.java:1264`

Code excerpt:

```java
public static void close() {
    System.out.println("Runtime cpu collector close success");
}
```

Root cause:
`close()` is called during server shutdown from `MultiWorkerServer.java:1264`. The method does nothing except print a message to `System.out`. It does not release the static `OperatingSystem os` field or any native resources held by OSHI. It also uses `System.out.println` instead of the SLF4J logger used throughout the rest of the codebase.

Impact:
Low — OSHI's `OperatingSystem` does not implement `Closeable` and does not require explicit cleanup in most versions. The `System.out.println` is a cosmetic inconsistency. The main concern is that the method name `close()` implies cleanup that does not actually happen, which could mislead future maintainers.

Suggested fix:
Either remove the `close()` method entirely (since it does nothing useful) or add a log statement using SLF4J:

```java
public static void close() {
    log.info("Runtime CPU collector closed");
}
```

If OSHI resource cleanup is needed in the future, this method should actually perform it.

## Reviewer Notes (AI agent 2)

Reviewer: AI agent 2
Date: 2026-05-20
Status: All three findings confirmed, with refinements for Finding 2 and Finding 3.

### Finding 1: Confirmed

`BigKeyTopK.BigKey` overrides `equals()` at `src/main/java/io/velo/monitor/BigKeyTopK.java:25-35` so equality is based only on
`key`. Because the record does not override `hashCode()`, the generated record hash still includes both `key` and `length`.
That makes `new BigKey("foo", 10).equals(new BigKey("foo", 20))` true while their hash codes can differ.

The current `BigKeyTopK.add()` path uses `PriorityQueue.removeIf()` and direct equality checks, so the current queue logic is
not broken by this. The object contract violation is still real and should be fixed before this record is reused in any
hash-based collection or assertion that relies on the standard Java equality contract.

### Finding 2: Confirmed with initialization-order refinement

Runtime updates are confirmed broken. `OneSlot.initBigKeyTopK(int)` at `src/main/java/io/velo/persist/OneSlot.java:812-817`
only creates a tracker when `bigKeyTopK == null`. A normal `OneSlot` constructor calls `initBigKeyTopK(10)` at
`src/main/java/io/velo/persist/OneSlot.java:198`, so later calls through `OneSlot.updateDynConfig()` and
`DynConfig.update()` persist and log the new `monitor_big_key_top_k` value but leave the existing tracker unchanged.

The same applies to `manage dyn-config`, which routes each slot through `oneSlot.updateDynConfig(configKey, configValue)` in
`dyn/src/io/velo/command/ManageCommand.groovy:898-902`. `DynConfig.applyUpdate()` uses the `SUPPORTED_ITEMS` path at
`src/main/java/io/velo/persist/DynConfig.java:432-438`, and the `BigKeyTopK` item logs success after calling the no-op guarded
initializer at `src/main/java/io/velo/persist/DynConfig.java:80-84`.

One refinement: persisted per-slot `dyn-config.json` values loaded during `OneSlot` construction can be applied before the
default `initBigKeyTopK(10)` call, because `DynConfig` is constructed at `src/main/java/io/velo/persist/OneSlot.java:141-142`
and applies file entries at `src/main/java/io/velo/persist/DynConfig.java:413-417`. The confirmed bug is therefore strongest
for runtime updates and `ConfForGlobal.initDynConfigItems` applied later in
`src/main/java/io/velo/persist/LocalPersist.java:213-216`, not for every persisted file reload path.

Fix note: do not compare the requested `k` with `bigKeyTopK.getQueue().size()`. Queue size is the current number of tracked
keys, not the configured capacity. The production fix should either expose/store the configured capacity and recreate only
when it changes, or always replace the tracker when the dynamic config is applied.

### Finding 3: Confirmed as a low-severity maintainability issue

`RuntimeCpuCollector.close()` at `src/main/java/io/velo/monitor/RuntimeCpuCollector.java:13-16` only prints
`Runtime cpu collector close success` to `System.out`. It is called from server shutdown at
`src/main/java/io/velo/MultiWorkerServer.java:1265`, but it does not release or reset anything.

I agree this is not a demonstrated resource leak because the OSHI `OperatingSystem` object used here does not expose a close
contract. The confirmed issue is that the method name and success message imply cleanup that is not performed, and the direct
`System.out.println` is inconsistent with the logger-based style used elsewhere.

## Review Feedback - Finding 3 Fix Commit `83d7848f`

Reviewed by: AI agent 2
Date: 2026-05-20

### Summary

Commit `83d7848f` replaces the direct `System.out.println()` in `RuntimeCpuCollector.close()` with SLF4J logging and adds a
test that asserts the old raw println message is no longer emitted.

### Blocking Concern

This fix should not be accepted as-is. `RuntimeCpuCollector.close()` is called from `MultiWorkerServer.onStop()` at
`src/main/java/io/velo/MultiWorkerServer.java:1265`, and that shutdown block intentionally uses `System.out.println()` for
other stop-stage messages at `src/main/java/io/velo/MultiWorkerServer.java:1216`, `1237`, `1240`, `1252`, `1255`, `1259`,
and `1262`. If Log4j output is unavailable or already shutting down during `onStop`, replacing the direct print with
`log.info()` can remove the shutdown-visible CPU collector close message entirely.

The original Finding 3 treated `System.out.println()` as a cosmetic inconsistency because no shutdown-specific reason had
been established. The current lifecycle context changes that assessment: direct stdout is consistent with the surrounding
shutdown path and appears intentional for visibility at stop time.

### Test Concern

`src/test/groovy/io/velo/monitor/RuntimeCpuCollectorTest.groovy:6-21` encodes the wrong requirement. It only verifies that
the old success string is absent from captured `System.out`, so it would also pass if `close()` emitted no observable
shutdown message at all. It does not simulate `onStop` or verify that a close marker remains visible when Log4j output is not
usable.

### Recommended Fix

Keep direct stdout for this shutdown-stage marker, but remove the misleading "success" wording and clarify that no OSHI
resource cleanup happens. For example:

```java
/** Emits a shutdown marker for the runtime CPU collector. */
public static void close() {
    System.out.println("Runtime cpu collector closed");
}
```

Alternatively, remove the method and print an explicit shutdown marker from `MultiWorkerServer.onStop()`, but do not replace
the only shutdown-visible message with logger-only output unless the stop-stage Log4j lifecycle is proven reliable.
